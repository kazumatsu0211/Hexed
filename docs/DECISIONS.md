## 2026-05-06: 関数の戻り値型注釈は export のみ必須

### 文脈
当初 CONVENTIONS §1 で「すべての関数の引数・戻り値に型を明示」と定めたが、
内部ヘルパー関数まで全注釈は冗長で、TypeScript の型推論の利点を活かせない。
Phase 1 の `src/core/board.ts` リファクタで `buildEffectPool`, `buildGrid`,
`shuffle` 等の内部ヘルパーが増え、規約と実装の整合性を見直す必要が生じた。

### 決定
- 引数の型：常に明示（推論不可のため）
- 戻り値の型：**export 関数のみ明示**、内部ヘルパーは推論に委ねる

### 理由
- export 関数 = モジュール境界 = 公開 API → 契約として明示すべき
- 内部ヘルパー = 同一ファイル内で完結 → 推論で十分、変更時の追従も容易
- 業界主流（Google TypeScript Style Guide,
  `@typescript-eslint/explicit-module-boundary-types`）と整合
- Unity 移植時、C# の public メソッドはシグネチャ明示が必須なので、
  export 注釈の習慣がそのまま活きる

### 代替案
- A. すべて注釈（当初 CONVENTIONS）→ 内部ヘルパーで冗長
- B. すべて推論 → API 契約が不明瞭、エラー位置がブレる

### トレードオフ
- 「export かどうか」の判断が一手間増える
- 将来 `@typescript-eslint/explicit-module-boundary-types` を導入すれば
  機械的に強制可能

## 2026-05-XX: Web プロトを先に作り、後で Unity に移植する

### 文脈
HEXED は最終的に Unity でリリースしたい（ポートフォリオ価値、Steam 公開の余地）。
しかし、いきなり Unity で作ると環境構築・学習コストが高く、ゲームデザインの
試行錯誤に時間がかかる。

### 決定
React + TypeScript で Web プロトを先に作り、面白さが検証できてから Unity へ
移植する2段階アプローチを採用。

### 理由
- Web は環境構築が早く、ホットリロードで試行錯誤しやすい
- ゲームロジック（`src/core/`）を React 非依存で書けば、C# への移植が機械的
- 完成した Web 版は itch.io で公開でき、それ自体がポートフォリオになる
- フィードバックを Web 版で集めてから Unity に投資できる

### 代替案
- A. 最初から Unity で作る → 環境構築の重さ、学習曲線が急
- B. Web のまま完成させる → Unity 経験が積めない、業界アピール弱い

### トレードオフ
- 同じゲームを2回作ることになる（実質的には移植なので30%）
- ただし、ゲームロジックが固まった状態で Unity 実装に入れるので効率的

## 2026-05-XX: 状態管理に Zustand を採用

### 文脈
HEXED はゲーム盤面、HP、アイテム、ターン等の複合的な状態を管理する。
React の `useState` だけでは扱いきれない。

### 決定
Zustand を採用。

### 理由
- Redux Toolkit より学習コスト低い、ボイラープレートが少ない
- Jotai のアトム単位より、ゲーム全体ステートに合う
- TypeScript との親和性が高い
- Unity 移植時に C# クラスへの翻訳が容易（中央集権的な状態 → ScriptableObject 等）

### 代替案
- A. Redux Toolkit → 過剰、学習コスト高
- B. Jotai → アトム単位がゲームには細かすぎる
- C. React Context のみ → 大規模になると性能と保守性に難

### トレードオフ
- ストアが大きくなった時にセレクタ最適化が必要
- 厳密な状態管理が必要な大規模アプリには向かない（が今回は問題なし）

### 参考
- 公式ドキュメント: https://docs.pmnd.rs/zustand

## 2026-05-XX: `src/core/` を React 非依存にする

### 文脈
将来 Unity（C#）に移植する計画がある。Web 版実装の中で React と密結合な
コードを書くと、移植時にすべて書き直しになる。

### 決定
ゲームロジックを `src/core/` に隔離し、ここでは React・DOM・ブラウザ API を
一切 import しない方針とする。

### 理由
- pure な TypeScript なら C# への翻訳が機械的に可能
- テストが書きやすい（依存なしで単体テスト可能）
- ロジックと UI が分離されることで保守性も向上

### 代替案
- A. 普通に React に依存して書く → 移植時にコスト爆発
- B. core を別パッケージに分ける（モノレポ化）→ オーバーキル

### トレードオフ
- 慣れるまでは「core で扱う型」「React で扱う型」を意識する必要あり
- 命名規則とディレクトリ構成の規律が必要
